# AI Coach — Техническое задание (детализированное)

> Документ описывает продукт «под ключ»: цели, стек, авторизацию, дизайн-систему, экраны, модель данных, API и пошаговую стратегию реализации. Цель документа — минимизировать отклонение реализации от требований. Документ задуман как минимальный, но цельный каркас, который можно точечно править и расширять.

## 1. Общее описание

**AI Coach** — модульное приложение для трекинга тренировок, питания, здоровья (анализы) и самочувствия (ощущения) с финальным слоем AI-аналитики. Главная ценность: пользователь видит актуальную картину своего состояния и получает персональные рекомендации, основанные на накопленных данных и поиске закономерностей между модулями.

**Ключевой принцип продукта:** минимум friction.
- Добавить тренировку — за ~30 секунд.
- Сфотографировать еду и получить расшифровку БЖУ — в 2 тапа.
- Внести настрой/самочувствие — за ~15 секунд.
- Увидеть сводку/прогресс — мгновенно при открытии.

**Модульность как принцип (см. п. 2.4 и п. 10):** приложение состоит из независимых модулей — **Тренировки**, **Питание**, **Анализы**, **Ощущения**. Каждый модуль самоценен и работает в отрыве от остальных; при наличии нескольких модулей система ищет кросс-модульные закономерности. Новый модуль можно добавить, не ломая существующие.

**Форм-фактор (см. п. 3 «Авторизация и точки входа»):** единое веб-приложение (SPA), работающее в двух режимах:
1. Как **Telegram Mini App** (WebView внутри Telegram) — основной канал привлечения и удержания, мобильный UX.
2. Как **обычный веб-сайт** (PWA) с входом по email или Google — для пользователей вне Telegram и для полноценной desktop-версии с крупными графиками.

**Целевая аудитория (расширенная):**
1. Тренирующиеся регулярно — ведут тренировки, питание, прогресс (от новичков до продвинутых).
2. **Люди без модуля тренировок** — хотят отслеживать анализы, питание и самочувствие за период, чтобы стыковать паттерны: «что поел → почему болит живот → что делать», «как сон/стресс влияют на самочувствие и показатели».

Оба сегмента обслуживаются одной системой за счёт модульности: пользователь включает только нужные модули, а аналитика работает на доступных данных.

## 2. Технологический стек и архитектура

### 2.1. Бэкенд

> Стек выбран как лёгкий, современный и высокопроизводительный, с сильной AI-составляющей (поиск паттернов — ядро ценности). Нагрузка преимущественно I/O-bound (БД + внешние AI-вызовы), поэтому ставка на async-Python даёт лучший баланс производительности и скорости разработки, чем компилируемые языки, и нативно закрывает анализ данных.

| Слой | Технология | Обоснование |
|------|-----------|-------------|
| Язык/Фреймворк | **Python 3.12 + FastAPI** | Async, высокая производительность (Starlette/Uvicorn), авто-OpenAPI, быстрая разработка |
| API | REST, JSON, версия `/api/v1`, авто-документация Swagger/OpenAPI | Простая интеграция с SPA и Mini App |
| Валидация/схемы | **Pydantic v2** | Быстрая валидация (ядро на Rust), контракты запросов/ответов |

| БД | **PostgreSQL 16** | JSONB для сырых ответов AI/анализов, надёжность, аналитические запросы |

## 3. Авторизация и точки входа

### 3.1. Три способа входа (требование №2)

| Способ | Канал | Поведение |
|--------|-------|-----------|
| **Telegram** | Mini App | Бесшовный вход по InitData при запуске из мини-аппа (нулевой friction) |
| **Email + пароль** | Web/PWA | Классическая регистрация и вход |
| **Google OAuth** | Web/PWA | Вход в один тап через Google-аккаунт |

Все три провайдера ведут к **единому пользователю** через таблицу `auth_identities` — аккаунты можно связать (один человек = один профиль).

### 3.2. Модель идентичностей

- `users` — ядро пользователя (профиль, параметры, цели, включённые модули).
- `auth_identities` — провайдеры: `telegram` | `email` | `google`. Поля: `provider`, `provider_id`, `email`, `password_hash` (для email), `meta` (jsonb: username/avatar/google sub и т. п.).

Один `user` может иметь несколько `auth_identities` (например, Telegram + Google + email в одном профиле).

### 3.3. Сценарии входа

1. **Запуск внутри Telegram (Mini App):**
   - Фронт берёт `initData` из Telegram SDK → `POST /api/v1/auth/telegram`.
   - Бэкенд валидирует подпись `initData` по `BOT_TOKEN` (HMAC-SHA256, проверка `hash` и `auth_date` ≤ 24ч).
   - Identity `telegram:<id>` найдена → вход; иначе создаём `user` + identity, подтягиваем имя/username/photo.
   - Возвращается пара JWT (access + refresh).

2. **Email:**
   - `POST /api/v1/auth/register` `{email, password, name}` → `user` + identity `email`, опц. письмо-подтверждение.
   - `POST /api/v1/auth/login` `{email, password}` → JWT.

3. **Google:**
   - Фронт получает `id_token`/`code` от Google → `POST /api/v1/auth/google` `{credential}`.
   - Бэкенд валидирует токен Google, извлекает `sub`/`email` → identity `google:<sub>`; вход или создание профиля.

4. **Связывание аккаунтов (после MVP):** залогиненный пользователь добавляет второй провайдер (`POST /api/v1/auth/link/{provider}`).

### 3.4. Онбординг (после первого входа) — оставлен (требование №5)

Короткий онбординг, адаптируется под цели и интересующие модули:
1. **Цели** (множественный выбор): Набор массы / Похудение / Выносливость / Гибкость / Здоровье.
2. **Какие модули нужны** (множественный выбор, можно изменить позже): Тренировки / Питание / Анализы / Ощущения. Влияет на состав навигации и аналитику.
3. **Параметры:** пол, вес, рост, возраст (нужны для формул целей, п. 9).
4. **Готово** — короткое пояснение интерфейса.

> Онбординг полезен: собранные на нём цели/параметры/выбранные модули позволяют строить зависимости и давать персонализированные данные. Для Telegram-канала имя/username/аватар предзаполняются автоматически.

### 3.5. Безопасность и приватность (требование №6)

- Telegram InitData валидируется на сервере (подпись + `auth_date` ≤ 24ч).
- Google `id_token` валидируется по публичным ключам Google (issuer/audience/exp).
- Email-пароли — хеш `argon2`/`bcrypt`.
- **Все данные строго scoped по `user_id`**: на уровне зависимостей FastAPI каждый запрос привязан к владельцу токена; запрос чужих данных невозможен.
- **Персональные AI-рекомендации и инсайты доступны только самому пользователю** — не агрегируются, не публикуются, не передаются третьим лицам.
- **Шифрование чувствительных полей at rest** (анализы/биомаркеры, заметки самочувствия) — application-level encryption ключом из секретов.
- Секреты — только через переменные окружения.
- Rate limiting на auth-эндпоинтах; HTTPS-only куки/заголовки.
- Полное удаление данных по запросу (cascade + удаление файлов из S3).

## 4. Дизайн-система и UX

### 4.1. Принципы

- Все ключевые действия — в **1–2 тапа**.
- **Нет длинных форм** — чипы, тоглы, степперы, быстрые селекторы.
- **Тёмная тема по умолчанию** (синхронизируется с темой Telegram через SDK), светлая — опционально.
- **Haptic-фидбек** при значимых действиях (Telegram SDK; в web — no-op).
- Простота и функциональность одинаково важны на mobile и desktop (см. п. 5).
- Тяжёлый выбор — **full-screen overlay** (Exercise Picker), а не модалки на пол-экрана.
- Списки — **бесконечная пагинация**, графики — **адаптивные**.
- **Офлайн-режим:** TanStack Query persist + очередь несинхронизированных мутаций.
- Скелетоны при загрузке, оптимистичные апдейты при сохранении.

### 4.2. Цвета модулей и психология (требование №8)

Каждому модулю — свой устойчивый акцент. Замечание про «жёлтый для Ощущений» учтено: жёлтый соседствует с оранжевым (оба тёплые, легко спутать с «Тренировками») и психологически тяготеет к тревоге/предупреждению при низком контрасте. Поэтому для Ощущений рекомендуется **фиолетовый** — два тёплых и два холодных акцента, равномерно разнесённые по кругу.

| Модуль | Цвет | Психология |
|--------|------|-----------|
| **Тренировки** | Оранжевый `#F97316` | Энергия, движение, физическая активность, мотивация |
| **Питание** | Зелёный `#22C55E` | Здоровье, свежесть, баланс, натуральность |
| **Анализы** | Синий `#3B82F6` | Доверие, клиническая точность, спокойствие |
| **Ощущения** | Фиолетовый `#8B5CF6` | Интроспекция, ментальное состояние, спокойствие |

> Если важно сохранить исходную идею с жёлтым — допустим тёплый янтарный `#F59E0B` (оптимизм), но тогда контраст с оранжевыми «Тренировками» нужно усиливать насыщенностью и формой. Рекомендация по умолчанию — фиолетовый.

Каждый модульный цвет имеет шкалу оттенков (для фона-карточки, бордера, текста, «свечения» активного модуля) в Tailwind-теме.

### 4.3. Логотип (требование №8)

Концепт — **связность модулей**: четыре фигуры (кольца/лепестки/сегменты) в четырёх модульных цветах, пересекающиеся в общем центре. Центр символизирует пользователя/единую картину, лепестки — независимые модули, сходящиеся в одну систему. Логотип масштабируется до моно-варианта (один цвет) для фавикона и Telegram.

### 4.4. Дизайн-направления (требование №7 — выбрать неизбитый)

Все три строятся на shadcn/ui (кастомизация токенов), различаются «характером»:

- **A. Calm Data / Editorial (рекомендуется):** много воздуха, крупные числа метрик, тонкая типографика (Inter/Geist + акцентный display-шрифт Satoshi), сдержанные модульные акценты. Премиально, спокойно, не приедается, идеально для данных и графиков.
- **B. Soft Neo-brutalism lite:** чёткие карточки, заметные модульные цвета, высокий контраст, «чанковые» элементы с тонкими бордерами. Характерно и узнаваемо, но требует дисциплины, чтобы не выглядеть грубо.
- **C. Glass + Depth (dark):** тёмная база с мягкими градиентами и модульным «свечением» карточек активного раздела, лёгкий glassmorphism на ключевых карточках. Современно, но риск выглядеть «трендово-одноразово».

**Рекомендация:** база — **A (Calm Data/Editorial)** с дозированным «свечением» активного модуля из **C**. Это даёт спокойный, читаемый и неизбитый интерфейс под насыщенные данными экраны.

### 4.5. Визуальные токены (Tailwind theme)

- Базовый фон: тёмно-нейтральный (`zinc/neutral-950..900`).
- Поверхности-карточки: на 1 тон светлее фона, `rounded-2xl`, мягкие тени.
- Акценты — четыре модульных цвета (п. 4.2); CTA берёт цвет активного модуля.
- Семантические цвета (нормы): зелёный (≥80% / норма), жёлтый (50–79%), красный (<50% / выход за референс).
- Типографика: системный/гротеск, крупные числа для метрик, мелкие подписи единиц.

### 4.6. lazyweb MCP

Для генерации полированных, неизбитых блоков (лендинг PWA, hero, пустые состояния, карточки сводки) используется **lazyweb MCP**, результат адаптируется под токены дизайн-системы и shadcn/ui.

## 5. Навигация: разделение mobile и desktop (требование №9)

Функционал **полностью дублируется** на mobile и desktop; различается только раскладка. Mobile — компактный мини-апп; desktop — широкая версия с крупными графиками/расчётами «в масштабе».

### 5.1. Mobile (Telegram Mini App / узкий экран) — Bottom Tab Bar

| Таб | Иконка (Lucide) | Название |
|-----|-----------------|----------|
| 1 | Home | Главная |
| 2 | Dumbbell | Тренировки |
| 3 | Utensils | Питание |
| 4 | HeartPulse | Ощущения |
| 5 | User | Профиль |

- Bottom Tab Bar всегда виден, с учётом safe-area; активный таб подсвечен цветом своего модуля.
- **Анализы** живут в Профиле (секция), а также в кросс-модульной аналитике; **Прогресс/графики** доступны внутри каждого модуля и на desktop-дашборде.
- Состав табов адаптируется под включённые модули (онбординг, п. 3.4): отключённый модуль скрывается.

### 5.2. Desktop (web/PWA, широкий экран)

- Левый **sidebar** с теми же разделами + раздел **Анализы** и расширенный **Прогресс**.
- Главная — широкий дашборд: графики/сводки в сетке, всё видно «в масштабе» без переходов.
- Те же действия (создать тренировку, добавить приём пищи, внести настрой) — из дашборда и из разделов.

Раскладка переключается адаптивно (один кодовый набор экранов, разные контейнеры).

### 5.3. Главная (общая логика)

Главная — **суммаризация всего**: краткая выжимка, чем живёт пользователь:
- количество тренировок (за период) и последняя тренировка;
- потребление калорий/БЖУ факт/цель;
- текущее настроение/самочувствие (тренд);
- прогресс по показателям (рекорды, баланс мышц, динамика веса, отклонения анализов).

С Главной можно **создать тренировку**, **добавить приём пищи**, **внести настрой** в один тап. В каждом разделе — подробная детализация (вся история активности модуля).

## 6. Детализация экранов

> Привязка к этапам — в дорожной карте (п. 13). Внутри экранов отмечено, что относится к ядру (без AI), а что — к AI-слою (этап 5).

### 6.1. Экран «Главная» (Dashboard)

**Назначение:** сводка состояния, быстрые действия, (этап 5) AI-рекомендация дня.

Секции сверху вниз (показываются по включённым модулям):

1. **Шапка:** дата (день недели, число, месяц); аватар + имя (тап → Профиль).
2. **AI-рекомендация дня** *(этап 5)* — карточка с акцентом, 1–2 предложения, иконка/цвет категории. До этапа 5 секция скрыта.
3. **Быстрые действия** (по модулям): «Начать тренировку», «Добавить питание», «Внести настрой».
4. **Сегодня: питание** *(этап 2)* — прогресс-бар калорий факт/цель, под ним Б/Ж/У.
5. **Баланс мышц за неделю** *(этап 1)* — бары по группам, цвет по проценту, «X/Y подходов» и %.
6. **Последняя тренировка** *(этап 1)* — дата, тип, самочувствие (emoji), число упражнений.
7. **Самочувствие** *(этап 3)* — подсказка заполнить утреннюю/вечернюю анкету + краткий тренд настроения.
8. **Анализы** *(этап 4)* — карточка последних показателей с отклонениями (если модуль включён).

**Пустые состояния:** карточки с призывом к действию (через lazyweb MCP).

### 6.2. Модуль «Тренировки» *(этап 1)*

#### 6.2.1. «Новая тренировка»

1. **Тип** — чипы (один выбор): Силовая, Кардио, Растяжка, HIIT, Комбинированная.
2. **Список упражнений** — раскрывающиеся карточки: название + «N подходов»; таблица подходов (Вес кг | Повторы | RPE опц. | удаление); «+ Подход» дублирует последний; порядок ↑↓.
3. **Самочувствие тренировки** — 5 emoji-кнопок (😞→🔥).
4. **Заметки** — textarea 1–3 строки.
5. **FAB «+ Упражнение»** → full-screen Exercise Picker (6.2.4).
6. **«Сохранить»**: валидация → `POST /workouts` → toast. Сервер авто-обновляет PR.

#### 6.2.2. «История тренировок» + Календарь (требование №11)

1. **Календарь** (`react-day-picker`): дни с тренировками помечены меткой (точка/бейдж цветом модуля «Тренировки»); тап по дню → тренировки этого дня.
2. **Итоги:** общее число тренировок **за месяц** и **за год** (переключение месяц/год листанием календаря), суммарный тоннаж за период.
3. Переключатель периода (чипы): Неделя | Месяц | 3 месяца | Всё время — фильтрует список под календарём.
4. Список карточек: дата + день недели, тип (badge), число упражнений, самочувствие (emoji), тоннаж.

#### 6.2.3. Детальный просмотр тренировки

Шапка (дата, тип, длительность, самочувствие), список упражнений с подходами, заметки, «Удалить» (с подтверждением).

#### 6.2.4. Exercise Picker (full-screen overlay)

1. Поиск по названию, debounce 200ms.
2. Фильтр-чипы категорий: Все | Грудь | Спина | Плечи | Руки | Ноги | Кор | Кардио | Всё тело.
3. Список: название, категория (мелко), кнопка «+» / «Добавлено».
4. «Своё упражнение»: форма (название, категория, **мышечные группы — мультивыбор**, п. 7) → `is_custom=true`.

### 6.3. Прогресс (внутри модулей / desktop-дашборд) *(этап 1; «Инсайты» — этап 5)*

Период (Неделя | Месяц | 3 месяца | Год) + вкладки-чипы: Объём | Рекорды | Мышцы | Инсайты.

- **Объём:** BarChart тоннажа; LineChart веса тела; LineChart самочувствия.
- **Рекорды:** карточки PR (упражнение, тип: макс. вес/повторы/объём, значение, дата). Авто-обновление на сервере.
- **Мышцы:** список групп (17), прогресс-бар факт/цель, «X/Y подходов» + %, цвет по проценту. *(этап 5)* AI-подсказка по дисбалансу.
- **Инсайты** *(этап 5)*: карточки с цветом по категории модуля.

### 6.4. Модуль «Питание» *(этап 2)*

#### 6.4.1. Основной вид

1. Чипы приёма: Завтрак | Обед | Ужин | Перекус.
2. **Карточка «Сфотографировать еду»**: тап → камера → `POST /nutrition/recognize` → спиннер → предзаполненные поля БЖУ → правка.
3. Список блюд: название, сетка 2×2 (Вес г, Белки, Жиры, Углеводы), авто-калории (Б×4 + Ж×9 + У×4).
4. «+ Добавить блюдо».
5. **Итого** (если ≥1 блюдо): Калории | Белки | Жиры | Углеводы.
6. «Сохранить приём пищи» → `POST /nutrition`.

#### 6.4.2. Дневная сводка (сворачиваемая, наверху)

Прогресс-бар калорий факт/цель; под чертой Б/Ж/У факт/цель.

#### 6.4.3. История питания

Список дней (итоговые калории и БЖУ); тап → развёрнутый вид по приёмам.

> Распознавание еды — самостоятельная утилита ввода (не аналитика). Ручной ввод обязателен и работает без AI.

### 6.5. Модуль «Ощущения» — анкеты 2×/день *(этап 3)*

**Назначение:** дважды в день фиксировать состояние, чтобы видеть динамику и (этап 5) искать паттерны (например, «что поел → как себя чувствую»).

- **Утренняя анкета:** Качество сна (1–5), Энергия при подъёме (1–5), Настроение (1–5), Мышечная болезненность (1–5), Мотивация (1–5). Опц.: вес тела, заметка.
- **Вечерняя анкета:** Энергия за день (1–5), Стресс (1–5), Настроение (1–5), Усталость (1–5), Удовлетворённость днём (1–5). Опц.: заметка.

UI: карточка с 5 кнопками на вопрос (один тап), ≤1 экрана, ~15–20 секунд. Доступ: таб «Ощущения» + подсказка на Главной. Одна утренняя и одна вечерняя запись в сутки (`slot`), редактирование в течение дня. Графики самочувствия — в Прогрессе.

### 6.6. Экран «Профиль» *(этап 1; «Анализы» — этап 4)*

1. **Шапка:** аватар, имя, username/email.
2. **Цели** (множественный выбор) — влияют на БЖУ/калории, целевые подходы, AI-рекомендации.
3. **Параметры:** пол, вес, рост, возраст — редактируемые (формулы — п. 9).
4. **Модули:** включение/отключение модулей (меняет навигацию и аналитику).
5. **Анализы** *(этап 4)*:
   - «Загрузить анализы» (PDF/фото) → `POST /health/recognize` → AI извлекает биомаркеры → правка → сохранение.
   - «Последний анализ» → детальный просмотр (биомаркеры: значение+единица, референс, статус 🟢/🔴/🔵; «Удалить»).
6. **Настройки:** единицы (кг/фунты), язык (заготовка), управление провайдерами входа (Telegram/email/Google), удаление всех данных (с подтверждением).

## 8. Серверная логика

### 8.1. Автоматические Personal Records *(этап 1)*

При сохранении тренировки (`workouts.service` → `PersonalRecordService`):
1. Для каждого упражнения ищем текущие PR: `max_weight`, `max_reps`, `max_volume` (объём = вес × повторы).
2. Если новый результат лучше — обновляем/создаём `personal_records`.
3. Выполняется в транзакции вместе с сохранением тренировки.

### 8.2. Расчёт баланса мышц *(этап 1)*

`MuscleBalanceService`:
1. Берём все подходы за последние 7 дней (eager load `exercise.muscle_groups`).
2. Суммируем подходы на каждую группу (**подход засчитывается каждой связанной группе**, п. 7.1).
3. Сравниваем с целевым числом (п. 9.4); `percentage = факт / цель × 100`, цвет по порогам.
4. Кэшируется в Redis (TTL до конца дня), инвалидируется при сохранении тренировки.

### 8.3. Распознавание еды *(этап 2)*

`POST /api/v1/nutrition/recognize` (multipart, фото):
1. Фото → S3/MinIO → `recognize_food` (ARQ) → base64 → OpenAI Vision.
2. Промпт: «Определи блюда, оцени вес, посчитай БЖУ».
3. Ответ: JSON `[{name, weight, protein, fat, carbs, calories}]`.
4. Фронт получает предзаполненные поля; пользователь правит → `POST /nutrition`.

### 8.4. Распознавание анализов *(этап 4)*

`POST /api/v1/health/recognize` (multipart, фото/PDF):
1. Файл → S3/MinIO → `recognize_health` (ARQ) → OpenAI Vision.
2. Промпт: «Извлеки показатели: название, значение, единица, референс, статус».
3. Ответ: JSON-массив биомаркеров → подтверждение/правка → `POST /health`.

### 8.5. AI-рекомендации и поиск паттернов *(этап 5)*

`GET /api/v1/ai/recommendations` и `generate_insights` (ARQ):

**Контекст в LLM (только по доступным модулям пользователя):**
- Цели и параметры.
- Тренировки за период (число, тип, самочувствие, тоннаж, баланс мышц).
- Питание (калории, белок, БЖУ).
- Ощущения (сон, энергия, настроение, стресс, усталость).
- Анализы (последние, отклонения).

**Поиск закономерностей (ядро этапа 5, требование №3):** кросс-модульная корреляция рядов (pandas/numpy) + интерпретация LLM, например:
- «что поел → почему болит живот / плохое самочувствие» (стыковка питания и ощущений/симптомов);
- падение тоннажа/настроения после дней с дефицитом сна/калорий;
- связь высокого стресса с пропусками тренировок;
- дисбаланс мышц; отклонения в анализах относительно нагрузки/питания.

**Формат:** JSON-массив `{title, body, category}`, 2–5 инсайтов. Рекомендация дня — короткая запись, кэшируется на день. **Видна только владельцу** (п. 3.5).

## 9. Формулы расчёта целей

### 9.1. BMR (Миффлин-Сан Жеор)

- Мужчины: `BMR = 10 × вес(кг) + 6.25 × рост(см) − 5 × возраст + 5`
- Женщины: `BMR = 10 × вес(кг) + 6.25 × рост(см) − 5 × возраст − 161`

### 9.2. Целевые калории (TDEE)

`TDEE = BMR × коэффициент активности`: 1.2 / 1.375 / 1.55 / 1.725 (по числу тренировок за 2 недели). Корректировка: Набор массы `+300`; Похудение `−500`.

### 9.3. Целевые макронутриенты

- **Белки:** Набор массы 2.0 г/кг; Похудение 2.2 г/кг; остальные 1.6 г/кг.
- **Жиры:** 1.0 г/кг.
- **Углеводы:** `(целевые калории − белки×4 − жиры×9) / 4`.

### 9.4. Целевые подходы на группу (в неделю)

| Группа | Набор массы | Поддержание | Остальные |
|--------|-------------|-------------|-----------|
| Крупные (грудь, спина, квадрицепсы, ягодицы) | 12–16 | 8–12 | 6–10 |
| Средние (плечи, бицепс, трицепс, бицепс бедра) | 8–12 | 6–10 | 4–8 |
| Мелкие (икры, предплечья, пресс, косые, поясница) | 6–8 | 4–6 | 2–4 |

Конкретное число — от частоты: 2 трен./нед → минимум; 4+ → максимум.

> Формулы реализуются в `app/services/targets.py` и покрываются unit-тестами (pytest).

## 10. Модель данных и модульность

### 10.1. ER-схема

```
users
├── auth_identities (1:N)       # провайдеры: telegram, email, google
├── workouts (1:N)              # модуль Тренировки
│   └── workout_exercises (1:N)
│       └── exercise_sets (1:N)
├── nutrition_logs (1:N)        # модуль Питание
│   └── food_entries (1:N)
├── health_analyses (1:N)       # модуль Анализы
│   └── biomarkers (1:N)
├── subjective_logs (1:N)       # модуль Ощущения (slot: morning/evening)
└── personal_records (1:N)

exercise_catalog (глобальный справочник)
```

### 10.2. Принцип модульности (требование №10)

- Каждый модуль — независимый набор таблиц, связанных только с `users` (нет жёстких связей между модулями). Любой модуль работает в отрыве от остальных.
- Кросс-модульная аналитика строится **поверх** через общий ключ `user_id` + `date` (временная ось), а не через схемные связи. Это позволяет коррелировать любые доступные модули.
- Новый модуль = новая папка `app/modules/<name>` + свои таблицы + `include_router`. Существующие модули не затрагиваются.
- На фронте модуль = свой роут/таб + цвет; включается/отключается флагом пользователя.

### 10.3. Поля сущностей (SQLAlchemy / Alembic)

- **User:** id, name, username?, gender(enum), goals(jsonb), enabled_modules(jsonb), weight, height, age, activity_level, avatar_url?, locale, units, timestamps.
- **AuthIdentity:** id, user_id, provider(enum: telegram|email|google), provider_id, email?, password_hash?, meta(jsonb), created_at.
- **Workout:** id, user_id, date, type(enum), duration?, feeling(1–5)?, notes?, timestamps.
- **WorkoutExercise:** id, workout_id, exercise_id, exercise_name, order.
- **ExerciseSet:** id, workout_exercise_id, weight, reps, rpe?, order.
- **NutritionLog:** id, user_id, date, meal_type(enum), photo_url?, timestamps.
- **FoodEntry:** id, nutrition_log_id, name, protein, fat, carbs, calories, weight.
- **HealthAnalysis:** id, user_id, date, source, photo_url?, raw_data(jsonb, шифруется), timestamps.
- **Biomarker:** id, analysis_id, name, value, unit, reference_min?, reference_max?, status(enum: normal|high|low).
- **SubjectiveLog:** id, user_id, date, slot(enum: morning|evening), sleep_quality?, energy?, mood?, stress?, fatigue?, soreness?, motivation?, satisfaction?, body_weight?, notes?, timestamps. Уникальный индекс `(user_id, date, slot)`.
- **PersonalRecord:** id, user_id, exercise_id, exercise_name, type(enum: max_weight|max_reps|max_volume), value, achieved_at. Уникальный индекс `(user_id, exercise_id, type)`.
- **ExerciseCatalog:** id, name, category(enum), muscle_groups(jsonb, **массив групп**), is_custom, created_by?, description?.

### 11.1. Авторизация

## 12. Нефункциональные требования

### 12.1. Производительность
- Первый осмысленный рендер < 1.5с на 3G.
- API-ответы < 300мс (кроме AI — асинхронно через ARQ; UI не блокируется, polling статуса).

### 12.2. Масштабирование
- MVP: монолит FastAPI + 1 ARQ-воркер (Docker).
- 1000+: read-реплика PostgreSQL, кэш справочников в Redis, несколько воркеров.
- 10k+: выделенный AI-воркер/сервис, CDN для статики и фото.

### 12.3. Данные и приватность
- Фото еды/анализов — в S3-совместимом хранилище (MinIO локально).
- Чувствительные поля (анализы, заметки) — шифрование at rest.
- Персональные рекомендации/инсайты — только владельцу.
- GDPR: полное удаление данных по запросу (cascade + удаление файлов).

### 12.4. Безопасность
- Валидация Telegram InitData и Google id_token на сервере.
- Хеширование паролей (argon2/bcrypt), throttle, опц. подтверждение email.
- Все эндпоинты scoped по `user_id` через зависимости FastAPI.
- Секреты — через переменные окружения.

### 12.5. Контейнеризация
- Запуск через Docker обязателен (dev и prod на одинаковых образах).
- `docker-compose` поднимает весь стек одной командой.

## 13. Пошаговая стратегия реализации (дорожная карта)

> Стратегия: сначала каркас и ядро **без AI**, затем расширение по модулям, AI — последним слоем поверх накопленных данных. Каждый модуль самоценен и поставляем отдельно (модульность).

### Этап 0 — Фундамент (1 неделя)
- Каркас: FastAPI (модульная структура) + React/Vite SPA, Tailwind + shadcn/ui, **Docker** (api, worker, PostgreSQL, Redis, MinIO, web), Alembic-миграции.
- Авторизация: Telegram InitData + email + Google OAuth, JWT, `users` + `auth_identities`.
- Онбординг (цели, выбор модулей, параметры), профиль, навигация mobile (bottom tabs) и desktop (sidebar), тёмная тема, цвета и логотип модулей.
- **CI/CD (п. 14):** базовый пайплайн GitHub Actions (lint + test + build), публикация образов в GHCR, автодеплой `main` на staging.
- **Критерий приёмки:** вход тремя способами; онбординг; пустые экраны-заглушки; всё поднимается через `docker-compose`; PR проходит CI, образ публикуется в реестр.

### Этап 1 — Тренировки и статистика (без AI) (2 недели) — ПРИОРИТЕТ
- Каталог упражнений (сидер 34, **мультигруппы**, кастомные), Exercise Picker.

### Этап 2 — Питание (2 недели)
- Приёмы пищи, ручной ввод БЖУ, авто-калории, дневная сводка, история, целевые БЖУ/калории (п. 9).

### Этап 3 — Ощущения (анкеты 2×/день) (1 неделя)
- Утренняя/вечерняя анкеты, ограничение «одна на слот», редактирование в течение дня, подсказки на Главной.

- **Критерий приёмки:** анкеты за ~15–20с; данные в графиках.

### Этап 4 — Анализы и здоровье (1.5 недели)

## 14. CI/CD и инфраструктура поставки

> Цель: код удобно собирается, тестируется, версионируется, хранится в реестре образов и автоматически доставляется на сервер. Любой коммит проходит проверки качества; релизы воспроизводимы и откатываемы.

### 14.1. Платформа и стратегия веток

- **CI/CD:** GitHub Actions (по умолчанию). Альтернатива — GitLab CI (те же стадии, синтаксис `.gitlab-ci.yml`).
- **Ветки:** trunk-based — основная ветка `main` (всегда деплоима) + краткоживущие feature-ветки и PR.
- **Защита `main`:** merge только через PR с зелёным CI; запрет прямого push.

### 14.2. Триггеры и окружения

| Событие | Действие |
|---------|----------|
| PR в `main` | `lint` + `test` (бэкенд и фронт), сборка образов без публикации (проверка сборки) |
| Push в `main` | полный пайплайн → публикация образов с тегами → деплой на **staging** |
| Тег `v*.*.*` (semver) | публикация релизных образов → деплой на **prod** (с ручным approve) |

Окружения: `staging` (автодеплой из `main`) и `prod` (по тегу, с подтверждением). Изоляция секретов на уровне GitHub Environments.

### 14.3. Стадии пайплайна

1. **lint:** бэкенд — `ruff check` + `mypy`; фронт — `eslint` + `tsc --noEmit`.
2. **test:** бэкенд — `pytest` (поднимается сервисный PostgreSQL+Redis в job); фронт — `vitest`/`tsc`. Падение тестов блокирует merge/деплой.
3. **build:** многоступенчатая сборка Docker-образов `api` и `web` (`docker buildx`, кэш слоёв через GHA cache/registry cache).
4. **push:** публикация образов в **GitHub Container Registry (GHCR)** с тегами: `sha-<git-sha>` (иммутабельный), `main` (latest для staging), `vX.Y.Z` (релиз). Воркер использует образ `api` с другой командой запуска.
5. **deploy:** доставка на сервер (см. 14.5).

### 14.4. Хранение образов и версионирование

- Реестр образов — **GHCR** (приватный): `ghcr.io/<org>/ai-coach-api`, `.../ai-coach-web`.
- Каждый образ помечается иммутабельным тегом `sha-<git-sha>` → точное соответствие коммита и артефакта, основа для откатов.
- Релизы — **semver** через git-теги `vX.Y.Z`.
- Очистка старых образов — retention-политика реестра (например, хранить последние N + все релизные).

### 14.5. Деплой и миграции

- **MVP:** сервер с Docker; деплой через `docker compose pull && docker compose up -d` нужных тегов (SSH-шаг в Actions или GitOps-агент типа Watchtower по релизному тегу).
- **Миграции БД:** отдельный шаг `alembic upgrade head` перед стартом нового `api` (one-off контейнер). Бэкенд должен быть совместим на 1 версию назад (expand/contract) для безопасных миграций.
- **Health-check:** `/health` эндпоинт; деплой считается успешным после прохождения проверки.
- **Откат:** повторный деплой предыдущего `sha-`/`vX.Y.Z` тега образа (+ обратная миграция при необходимости).
- **Рост:** при переходе на Kubernetes стадии CI не меняются; меняется только шаг `deploy` (Helm/манифесты, rolling update).

### 14.6. Секреты и конфигурация

- Секреты (BOT_TOKEN, OPENAI_API_KEY, JWT-ключи, доступ к БД/S3, ключ шифрования полей) — в **GitHub Secrets** по окружениям, прокидываются как env в деплой.
- В репозитории — только `.env.example` (без значений). `.env` не коммитится.
- Образы не содержат секретов (передаются рантайм-окружением).

### 14.7. Поддерживаемость

- **Dependabot/Renovate** — авто-PR на обновление зависимостей (Python, npm, Docker base images).
- **Покрытие тестами** ключевой бизнес-логики (формулы целей, PR, баланс мышц) — обязательный gate.
- **Pre-commit hooks** (ruff/eslint) — локально, чтобы CI не падал на стиле.
- Бэкапы PostgreSQL (периодический дамп в S3) и версионирование схемы через Alembic.

## 15. Глоссарий

- **Модуль** — независимый домен (Тренировки/Питание/Анализы/Ощущения), включаемый отдельно.
- **Тоннаж** — сумма `вес × повторы` по всем подходам.
- **PR (Personal Record)** — личный рекорд по упражнению.
- **Баланс мышц** — соотношение фактических и целевых подходов на группу за неделю (подход засчитывается каждой связанной группе).
- **Слот анкеты** — `morning`/`evening`.
- **InitData** — подписанные данные пользователя из Telegram Mini App для валидации на сервере.
- **Кросс-модульная аналитика** — поиск закономерностей между модулями по общей оси `user_id` + `date`.
